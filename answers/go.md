## <a name="strings"></a>Строки

В Go строка в действительности является слайсом байт, доступным **только для чтения**. Строка содержит произвольные байты, и у неё нет ёмкости `cap`.

Создание подстрок работает очень эффективно. Поскольку строка предназначена только для чтения, исходная строка и строка, полученная в результате операции среза, могут безопасно совместно использовать один и тот же массив:

```Golang
var (
    str = "hello world"
    sub = str[0:5]
    usr = "/usr/kot"[5:]
)

print(sub, " ", usr) // hello kot
```

Конструкция `for ... range` итерирует строку посимвольно (а не побайтово, как можно было бы предположить):

```Golang
var str = "привет"

println(str, len(str)) // привет 12

for i, c := range str {
    println(i, c, string(c))
}

// 0 1087 п
// 2 1088 р
// 4 1080 и
// 6 1074 в
// 8 1077 е
// 10 1090 т
```

Встроенная функция `len()` возвращает количество **байт**

Чтобы определить **фактическую длину строки**, используйте метод из пакета `utf8`:

```Golang
utf8.RuneCountInString(str)
```

или преобразуйте массив из слайса байт в слайс `rune`:

```Golang
len([]rune(str))
```

Преобразование типов между `int` и `string` вида `string()` и `int()` будет невозможно. Для преобразования необходимо использовать функции из пакета `strconv` стандартной библиотеки go.

```Golang
num := 42
str := strconv.Itoa(num)
fmt.Println("Число в строку:", str)
```

## <a name="array"></a>Массивы

Массивы:

- Массивы — это упорядоченные структуры данных последовательностей элементов.
- Присваивание одного массива другому копирует все элементы.
- В частности, если передать массив в функцию, она получит копию массива, а не указатель на него.
- Массив имеет статический тип выделения памяти: после объявления изменить размер массива уже нельзя.
- Размер массива является частью его типа. Типы [10]int и [20]int различны.

```Golang
func Sum(a *[3]float64) (sum float64) {
  for _, v := range *a {
    sum += v
  }
  return
}
array := [...]float64{7.0, 8.5, 9.1}
x := Sum(&array) // Обратите внимание на явное использование оператора взятия адреса
```

Но даже этот стиль не является идиоматичным для Go. **Вместо этого используйте слайсы (срезы)!!!.**

## <a name="slice"></a>Слайсы (Срезы, Slices)

Слайс — это своего рода массив, но с динамическим выделением памяти.

```Golang
var foo []int
```

Когда вы создаёте новый слайс, рантайм создаст новую переменную этого типа, с нулевым указателем (nil), длиной и ёмкостью равными нулю.

```Golang
foo = make([]int, 5)
```

Эта команда создаст сначала массив из 5 элементов (выделит память и заполнит их нулями), установит значения `len` и `cap` в 5.

`Cap` означает ёмкость и помогает зарезервировать место в памяти на будущее, чтобы избежать лишних операций выделения памяти при росте слайса. Можно использовать другую аннотацию `make([]int, len, cap)`

```Golang
foo = make([]int, 3, 5)
```

Срезы содержат ссылки на лежащие в основе массивы, и если присвоить один срез другому, оба будут ссылаться на
один и тот же массив.

---

### Добавление к слайсу `append()`

`append()` — добавляет значение в слайс **и возвращает новый слайс**. Но не все так просто. Взглянем на следующий код:

```Golang
a := make([]int, 32)
a = append(a, 1)
```

`cap` — ёмкость означает количество выделенного места для массива. Функция `append()` проверяет, достаточно ли у слайса места, чтобы добавить туда ещё элемент, и если нет, то выделяет больше памяти.

---

Выделение — дорогая операция, поэтому `append()` пытается оптимизировать это, и запрашивает в данном случае памяти не для одной переменной, а для ещё 32х, — в два раза больше — чем начальный размер.

Если в слайсе `capacity` элементов было меньше 256, то вместимость увеличится в два раза. Если больше 256, то на четверть, эта штука называется _"классы размеров памяти"_.

**Выделение памяти пачкой один раз дешевле, чем много раз по кусочкам!**

Выделение памяти обычно означает выделение её по другому адресу и перемещение данных из старого места в новое. Это означает, что адрес массива, на который ссылается слайс также изменится. Возможен такой прикол:

```Golang
a := make([]int, 32)
b := a[1:16]
a = append(a, 1)
a[2] = 42
```

Мы получим два различных массива, и два слайса будут указывать на совершенно разные участки памяти! Если вы работаете с `append()` и подслайсами — будьте осторожны и имейте ввиду эту особенность.

---

### **ВАЖНО!!!**

Если слайс не инициализирован, функция `append()` самостоятельно инициализирует эту память, но только для одного элемента, с длиной и емкостью равной единице, соответственно. Такой подход не валиден, поскольку, если нам надо запихнуть большое количество элементов, каждый раз будет происходить выделение памяти, что очень дорого и не целесообразно. Гораздо лучше, заранее, с помощью функции `make()` инициализировать слайc и прописать длину и емкость.

## <a name="for"></a>Цикл for

В Go существует **только один** цикл — `for`, но он заменяет собой все остальные (while, do-while, foreach).

#### 1. Основные формы записи
*   **Классический:** `for i := 0; i < 10; i++ { ... }`
*   **Как `while`:** `for condition { ... }` (пока условие истинно).
*   **Бесконечный:** `for { ... }` (выход через `break` или `return`).

#### 2. Итерация по коллекциям (`for range`)
Используется для перебора массивов, слайсов, строк, мап и каналов.
```go
for index, value := range collection {
    // index — индекс (или ключ в map)
    // value — копия значения элемента
}
```
*   Если индекс не нужен: `for _, value := range ...`
*   Если нужно только значение (для каналов) или только индекс: `for i := range ...`

#### 3. Управление циклом
*   **`continue`**: Пропускает текущую итерацию и переходит к следующей.
*   **`break`**: Немедленно прерывает цикл.
*   **Метки (Labels)**: Позволяют выйти из вложенных циклов.
    ```go
    OuterLoop:
        for i := 0; i < 5; i++ {
            for j := 0; j < 5; j++ {
                if i == 2 && j == 2 {
                    break OuterLoop // Выйдет сразу из обоих циклов
                }
            }
        }
    ```

#### ⚠️ Важный нюанс: `range` и копирование (Частый вопрос на собеседовании)

**Проблема:** Когда вы используете `for index, value := range slice`, переменная `value` — это **копия** элемента слайса. Изменение `value` никак не повлияет на оригинальный слайс.

```go
nums := []int{1, 2, 3}
for _, v := range nums {
    v = v * 10 // Изменяется локальная переменная v, а не nums[i]
}
// nums все еще [1, 2, 3]
```

**Как правильно изменять значения в слайсе:**
Чтобы изменить оригинальный элемент, нужно обращаться к нему **по индексу**:

```go
nums := []int{1, 2, 3}
for i := range nums {
    nums[i] = nums[i] * 10 
}
// nums теперь [10, 20, 30]
```

## <a name="maps"></a>Map (Not actual since 1.24)

Карта (map или hashmap) — это неупорядоченная коллекция пар вида ключ-значение.

Мапы всегда передаются по "ссылке". В качестве ключей используются **сравниваемые типы**:

- Примитивные типы

  - `bool`

  - Числовые типы: `int`, `int8`, `uint8` ...

  - Типы с плавающей точкой: `float32`, `float64`

  - Комплексные числа: `complex64`, `complex128`

- Строки

- Указатели

- Массивы (**НЕ СЛАЙСЫ!!!**)

  - Массивы сравнимы, если их элементы сравнимы.

- Структуры

  - Структуры сравнимы, если все их поля являются сравнимыми типами.

- Интерфейсы

  - Интерфейсы сравнимы, если они содержат значения сравнимых типов.

---

```Golang
var m = make(map[string]int) // инициализация

m["one"] = 1 // запись в мапу

fmt.Println(m["one"], m["two"]) // 1 0
```

**Лучше выделить память заранее (передавая вторым аргументом функции make), если известно количество элементов — избежим эвакуаций!**

Для проверки существования ключа используем конструкцию вида (доступ к элементу карты может вернуть два значения вместо одного) называемую **"multiple assignment"**:

```Golang
var m = map[string]int{"one": 1}

v1, ok1 := m["one"] // чтение
v2, ok2 := m["two"]

fmt.Println(v1, ok1) // 1 true
fmt.Println(v2, ok2) // 0 false

for k, v := range m { // итерация всех эл-ов мапы
    fmt.Println(k, v)
}

delete(m, "one") // удаление

v1, ok1 = m["one"]

fmt.Println(v1, ok1) // 0 false
```

---

Упрощенный вариант работы hashmap:

- Передаем ключ "The Matrix" в хэш-функцию. Получаем `uint64` - 18002143618149980261;

- Вычисляем маску для наших бакетов. Она равна n-1, где n - количество бакетов. В примере 4 бакета, а маска равна 3;

- Вычисляем номер бакета, в котором сохраним наше значение. Для этого используем битовое **"и"**: `hash & mask == 18002143618149980261 & 3 == 01 & 11(отбросили нули) = 01`, что равно 1 в десятичной системе счисления;

- Идем в бакет по индексу 1 и перебором проверяем массив на наличие нашего ключа. Если находим совпадение по ключу, то перезаписываем значение, иначе добавляем в первое свободное место;

Особенности:

- В качестве ключа можно использовать любой тип данных для которого определена операция сравнения;

- При отсутствии элемента возвращается нулевое значение для типа. Вторым параметром можно получить флаг наличия элемента по ключу;

- Нельзя получить адрес элемента. Потому что при росте мапы оно переедет в другой бакет и адрес у него поменяется;

- Мапа подвержена гонке. Для конкурентного доступа можно использовать обертку из `sync.Map` или `мьютекс`;

- Порядок итерации не сохраняется. Под капотом каждый раз выбирается рандомный бакет, с которого начинается итерация;

- При каждом создании мапы генерируется seed для рандомизации хэш-функции. Это сделано для безопасности, так как зная хэш-функцию можно подобрать такие ключи, чтобы все значения попали в один бакет и мы получим линейную скорость поиска;

- При коллизиях используется стратегия `сlosed addressing`. Мы перебираем все ячейки бакета (их 8) и ищем первое свободное место;

- `OverloadFactor` равен 6.5 (~81% заполненности бакетов). Когда бакеты в среднем заполнены больше чем на 6.5 элементов, тригерится рост мапы, и все элементы перемещаются в новые бакеты, которых создается в два раза больше.

- При эвакуации элементы переносятся в новые бакеты постепенно.

Поиск может быть дороже чем `O(1)`, так что иногда выгоднее использовать бинарный поиск по слайсу, нежели, чем поиск в мапе.

### Горутины

Горутины (goroutines) представляют параллельные операции, которые могут выполняться независимо от функции, в которой они запущены. Главная особенность горутин состоит в том, что они могут выполняться параллельно и в разных потоках.

Их также называют легковесными потоками, потому что они управляются рантаймом языка, а не операционной системой.

Для запуска функции как горутину необходимо написать `go func()` , где `func()` - функция, которую хотите запустить.

```Golang
func main() {
    // цикл с 5-ю итерациями
	for i := 0; i < 5; i++ {
        // вызываем функцию greeting() как горутину
		go greeting()
	}
        // ожидаем незавершившиеся горутины
	time.Sleep(2 * time.Second)
}

func greeting() {
	fmt.Println("Hello World!!!")
}
```

_Минимальный (начальный) размер стэка составляет 2 КБ. Максимальный размер стэка горутины зависит от архитектуры системы и равен 1 ГБ для 64-разрядной архитектуры, 250 МБ для 32-разрядной архитектуры._

### Каналы

Каналы (channels) представляют инструменты коммуникации между горутинами. Примитив синхронизации.

Каналы в Go считаются блокирующими. Это означает, что любая горутина, которая пытается записать в заполненный канал, будет ждать, пока канал не опустеет, а любая горутина, которая пытается прочитать из пустого канала, будет ждать, пока в него не будет помещен хотя бы один элемент.

Каналы представлют структуру:

- qcount — количество элементов в буфере
- dataqsiz — размерность буфера
- buf — указатель на буфер для элементов канала (для хранения используется стэк)
- closed — флаг, указывающий, закрыт канал или нет
- recvq — указатель на связанный список горутин, ожидающих чтения из канала
- sendq -указатель на связанный список горутин, ожидающих запись в канал
- lock — мьютекс для безопасного доступа к каналу

По моей практике могу сказать, что меня спрашивали только о внутреннем устройстве каналов. Список из чего он состоит выше.

_В остальном, если интересно копнуть чуть глубже, советую [почитать](https://habr.com/ru/articles/308070) это._

#### Небуфферизированные каналы

Для создания небуферизированного канала вызывается функция make() без указания емкости канала:

```Golang
var intCh chan int = make(chan int) // канал для данных типа int
strCh := make(chan string)  // канал для данных типа string
```

_Горутина-отправитель может отправлять данные только в пустой канал. Горутина-получатель блокируется до тех пор, пока данные из канала не будут получены._

#### Буферизированные каналы

Мы также можем создавать буферизованные каналы, которые являются каналами, которым присваивается емкость, когда они создаются. Это означает, что даже если на канале не выполняется чтение, goroutine все равно может выполнять n записей, где n — емкость буферизованного канала. Вот как объявить и создать экземпляр:

```Golang
var intCh chan int = make(chan int, 4) // канал для данных типа int
strCh := make(chan string, 4)  // канал для данных типа string
```

#### Однонаправленные каналы

- Определение канала только для отправки данных:

```Golang
var inCh chan<- int
```

- Определение канала только для получения данных:

```Golang
var outCh <-chan int
```

#### Второе возвращаемое значение

Второе возвращаемое значение — это способ для операции чтения указать, было ли чтение из канала значением, сгенерированным записью в другом месте процесса, или значением по умолчанию, сгенерированным из закрытого канала. Секундочку; закрытый канал, что это?

#### Закрытие канала

В программах очень полезно иметь возможность указать, что больше никаких значений не будет отправлено по каналу. Это помогает процессам, идущим вниз по течению, знать, когда двигаться дальше, выходить, повторно открывать связь по новому или другому каналу и т. д. Мы могли бы сделать это с помощью специального значения-сигнала для каждого типа, но это удвоило бы усилия для всех разработчиков, и это на самом деле функция канала, а не типа данных, поэтому закрытие канала похоже на универсальный сигнал, который говорит: «Эй, восходящий поток больше не будет записывать никаких значений, делайте, что хотите». Чтобы закрыть канал, мы используем ключевое слово close, например:

```Golang
valueStream := make(chan interface{})
close(valueStream)
```

Интересно, что мы можем спокойно читать из закрытого канала. Например:

```Golang
intStream := make(chan int)
close(intStream)
integer, ok := <- intStream
fmt.Printf("(%v): %v", ok, integer)
```

Обратите внимание, что мы ничего не помещали в этот канал; мы закрыли его немедленно. Мы все еще могли выполнять операцию чтения, и фактически мы могли продолжать выполнять чтения в этом канале бесконечно, несмотря на то, что канал оставался закрытым. 

Как известно, для горутины выделяется стек, который стартует с 2 Кбайт и, на 64-битной архитектуре, может достигнуть 1 Гбайта. Утечка горутин приводит, как минимум, к утечке по памяти.

---

Ключевое слово range, используемое в сочетании с оператором for, поддерживает каналы в качестве аргументов и автоматически прерывает цикл при закрытии канала. Это позволяет выполнять краткую итерацию по значениям на канале. Давайте рассмотрим пример:

```Golang
intStream := make(chan int)
go func() {
   defer close(intStream) // 1
   for i := 1; i <= 5; i++ {
    intStream <- i
   }
}()

for integer := range intStream { // 2
   fmt.Printf("%v ", integer)
}
```
1. Здесь мы гарантируем, что канал закрыт, прежде чем мы выйдем из goroutine. Это очень распространенный шаблон.
2. Здесь мы работаем по intStream.

Обратите внимание, что циклу не нужны критерии выхода, а диапазон не возвращает второе логическое значение. Специфика обработки закрытого канала управляется для того, чтобы цикл был кратким.

#### Оператор select

Давайте начнем с того, что просто изложим один. Вот очень простой пример:

```Golang
var c1, c2 <-chan interface{}
var c3 chan<- interface{}
select {
   case <- c1:
   // Do something
   case <- c2:
   // Do something
   case c3<- struct{}{}:
   // Do something
}
```

Немного похоже на блок switch, не правда ли? Так же, как и блок switch, блок select охватывает ряд операторов case, которые защищают ряд операторов; однако, на этом сходство заканчивается. В отличие от блоков switch, операторы case в блоке select не проверяются последовательно, и выполнение не будет автоматически провалено, если ни один из критериев не будет выполнен.

Вместо этого все операции чтения и записи каналов рассматриваются одновременно, чтобы увидеть, готовы ли какие-либо из них: заполненные или закрытые каналы в случае чтения и каналы, которые не заполнены в случае записи. Если ни один из каналов не готов, весь оператор select блокируется. Затем, когда один из каналов будет готов, эта операция будет продолжена, и соответствующие ей операторы будут выполнены. Давайте рассмотрим быстрый пример:

```Golang
start := time.Now()
c := make(chan interface{})

go func() {
   time.Sleep(5*time.Second)
   close(c) // 1
}()

fmt.Println("Blocking on read...")
select {
   case <-c: // 2
    fmt.Printf("Unblocked %v later.\n", time.Since(start))
}
```

1. Здесь мы закрываем канал после пяти секунд ожидания.
2. Здесь мы пытаемся прочитать канал. Обратите внимание, что при написании этого кода нам не необходим оператор select — мы могли бы просто написать <-c — но мы расширим этот пример.

Результат:

```
Blocking on read...
Unblocked 5.000170047s later.
```

Как вы можете видеть, мы разблокируемся только примерно через пять секунд после входа в блок выбора. Это простой и эффективный способ блокировки, пока мы ждем, что что-то произойдет, но если мы задумаемся на мгновение, то можем придумать несколько вопросов:

- Что происходит, когда нескольким каналам нужно что-то прочитать?
- Что, если никогда не будет готовых каналов?
- Что, если мы хотим что-то сделать, но в данный момент ни один канал не готов?

Первый вопрос о том, что несколько каналов готовы одновременно, кажется интересным. Давайте просто попробуем и посмотрим, что произойдет.

```Golang
c1 := make(chan interface{}); close(c1)
c2 := make(chan interface{}); close(c2)

var c1Count, c2Count int
for i := 1000; i >= 0; i-- {
   select {
   case <-c1:
    c1Count++
   case <-c2:
    c2Count++
   }
}

fmt.Printf("c1Count: %d\nc2Count: %d\n", c1Count, c2Count)
```

Результат:

```
c1Count: 505
c2Count: 496
```

Как вы можете видеть, за тысячу итераций примерно в половине случаев оператор select считывает данные из c1, а примерно в половине случаев — из c2. Среда выполнения Go выполнит псевдослучайный равномерный выбор по набору операторов case. Это просто означает, что из вашего набора операторов case каждый имеет равные шансы быть выбранным, как и все остальные.

А как насчет второго вопроса: что произойдет, если никогда не будет каналов, которые будут готовы? Если нет ничего полезного, что вы можете сделать, когда все каналы заблокированы, но вы также не можете блокировать вечно, вы можете захотеть сделать тайм-аут. Пакет time Go предоставляет элегантный способ сделать это с каналами, который хорошо вписывается в парадигму операторов select. Вот пример использования одного из них:

```Golang
var c <-chan int
select {
case <-c: // 1
case <-time.After(1 * time.Second):
    fmt.Println("Timed out.")
}
```

1. Это утверждение case никогда не будет разблокировано, потому что мы читаем из нулевого канала

Функция time.After принимает аргумент time.Duration и возвращает канал, который будет отправлять текущее время после указанного вами периода

Это оставляет нам оставшийся вопрос: что происходит, когда ни один канал не готов, и нам нужно что-то сделать в это время? Как и операторы case, оператор select также допускает предложение по умолчанию на случай, если вы хотите что-то сделать, если все каналы, которые вы выбираете, блокируются. Вот пример:

```Golang
start := time.Now()
var c1, c2 <-chan int
select {
case <-c1:
case <-c2:
default:
    fmt.Printf("In default after %v\n\n", time.Since(start))
}
```

Это выведет: `In default after 1.421ms`

Вы можете видеть, что он выполнил оператор по умолчанию почти мгновенно. Это позволяет вам выйти из блока select без блокировки. Обычно вы увидите предложение по умолчанию, используемое в сочетании с циклом for-select. Это позволяет горутине продолжать работу, ожидая, пока другая горутина сообщит о результате. Вот пример этого:

```Golang
done := make(chan interface{})
go func() {
   time.Sleep(5*time.Second)
   close(done)
}()

workCounter := 0
loop:
for {
   select {
   case <-done:
     break loop
   default:
   }
   
   // Simulate work
   workCounter++
   time.Sleep(1*time.Second)
}
fmt.Printf("Achieved %v cycles of work before signalled to stop.\n", workCounter)
```

Вывод: `Achieved 5 cycles of work before signalled to stop.`

В этом случае у нас есть цикл, который выполняет какую-то работу и время от времени проверяет, следует ли его остановить. Наконец, есть особый случай для пустых операторов select: операторы select без предложений case. Они выглядят так:

```Golang
select {}
```

Это утверждение просто заблокирует навсегда.

### Cond

`sync.Cond` в языке Go — это **переменная условия** (condition variable). Она предназначена для сценариев, где несколько горутин должны ждать наступления определенного события или изменения состояния, прежде чем продолжить выполнение.

Простыми словами: `sync.Cond` позволяет горутинам «спать» и не тратить ресурсы процессора до тех пор, пока другая горутина не подаст сигнал о том, что условие выполнено.

---

### Основные методы

1.  **`Wait()`**: Горутина «засыпает». Она освобождает мьютекс и ждет сигнала. Когда горутина просыпается, она снова захватывает мьютекс перед выходом из метода.
2.  **`Signal()`**: Разблокирует (пробуждает) **одну** горутину, которая ждет на этом условии.
3.  **`Broadcast()`**: Разблокирует **все** горутины, которые ждут на этом условии.

---

### Правила использования (Важно!)

При работе с `sync.Cond` нужно соблюдать два строгих правила:

1.  **Всегда использовать `Locker` (обычно `sync.Mutex`):** `Wait()` должен вызываться при заблокированном мьютексе.
2.  **Вызывать `Wait()` только внутри цикла `for`:** Это защищает от «ложных пробуждений» (spurious wakeups) или ситуаций, когда другая горутина успела изменить состояние до того, как проснувшаяся горутина захватила мьютекс.

---

### Пример использования

Представим очередь с ограниченным размером. Читатели должны ждать, пока в очереди появится хотя бы один элемент.

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

type SharedResource struct {
	mu    sync.Mutex
	cond  *sync.Cond
	ready bool
}

func main() {
	res := &SharedResource{}
	res.cond = sync.NewCond(&res.mu)

	// Горутины-потребители
	for i := 1; i <= 3; i++ {
		go func(id int) {
			res.mu.Lock()
			// 1. Ждем условия в цикле for
			for !res.ready {
				fmt.Printf("Горутина %d ждет...\n", id)
				res.cond.Wait() 
			}
			fmt.Printf("Горутина %d приступила к работе!\n", id)
			res.mu.Unlock()
		}(i)
	}

	time.Sleep(2 * time.Second)

	// Горутина-производитель
	res.mu.Lock()
	res.ready = true
	fmt.Println("Производитель: данные готовы, бужу всех!")
	res.cond.Broadcast() // 2. Оповещаем всех
	res.mu.Unlock()

	time.Sleep(1 * time.Second)
}
```

### Когда НЕ стоит использовать `sync.Cond`?

*   Если вам нужно просто передать данные от одной горутины к другой — используйте **каналы**.
*   Если вам нужно просто подождать завершения группы горутин — используйте **`sync.WaitGroup`**.
*   Если вам нужно защитить доступ к переменной — используйте **`sync.Mutex`**.


### Pool

`sync.Pool` в Go — это инструмент для **повторного использования объектов**, который помогает снизить нагрузку на сборщик мусора (Garbage Collector, GC) и повысить производительность приложения.

### Для чего он нужен?

В Go создание (аллокация) объектов и их последующая очистка сборщиком мусора стоят ресурсов процессора. Если ваше приложение создает тысячи временных объектов в секунду (например, буферы для чтения сетевых пакетов или JSON-структуры), GC будет тратить много времени на их удаление.

`sync.Pool` позволяет:
1.  **Сохранить** объект после использования вместо того, чтобы отдать его GC.
2.  **Достать** уже готовый объект из пула, когда он понадобится снова, вместо создания нового.

У `sync.Pool` всего три ключевых элемента:

1.  **Поле `New`**: Это функция, которую вы определяете сами. Она вызывается, если вы пытаетесь достать объект из пула (`Get`), а там пусто. Она должна возвращать новый экземпляр объекта.
2.  **Метод `Get()`**: Возвращает объект из пула. Если в пуле есть свободный объект — отдает его, если нет — вызывает функцию `New`.
3.  **Метод `Put(x)`**: Возвращает объект обратно в пул, чтобы его мог использовать кто-то другой.

---

### Пример использования

Самый частый пример — использование буферов байтов (`bytes.Buffer`), чтобы не выделять память при каждой обработке строки.

```go
package main

import (
	"bytes"
	"fmt"
	"sync"
)

// Создаем пул буферов
var bufferPool = sync.Pool{
	New: func() any {
		fmt.Println("Создаем новый буфер")
		return new(bytes.Buffer)
	},
}

func main() {
	// 1. Достаем буфер из пула
	buf := bufferPool.Get().(*bytes.Buffer)
	
	// 2. Используем его
	buf.WriteString("Привет, ")
	buf.WriteString("Мир!")
	fmt.Println(buf.String())

	// 3. СБРАСЫВАЕМ состояние (важно!) и возвращаем в пул
	buf.Reset()
	bufferPool.Put(buf)

	// 4. Достаем снова — теперь создание нового не потребуется
	buf2 := bufferPool.Get().(*bytes.Buffer)
	fmt.Println("Достали буфер из пула, создание не вызывалось")
	bufferPool.Put(buf2)
}
```

---

### Важные особенности (Нюансы)

1.  **Автоматическая очистка**: Объекты в `sync.Pool` могут быть удалены сборщиком мусора в любой момент (обычно это происходит при каждом цикле GC). Пул — это не кэш в памяти (как Redis), вы не можете гарантировать, что объект там останется.
2.  **Очистка состояния**: Перед тем как положить объект обратно в пул через `Put()`, его **обязательно** нужно очистить (обнулить поля, вызвать `Reset()`). Иначе следующая горутина, которая возьмет этот объект, получит «грязные» данные от предыдущего использования.
3.  **Потокобезопасность**: `sync.Pool` полностью безопасен для использования из множества горутин одновременно.

---

### Когда стоит использовать `sync.Pool`?

*   Когда у вас есть **высокая нагрузка** и часто создаются однотипные временные объекты.
*   Когда профилирование (бенчмарки) показывает, что программа тратит много времени на `runtime.mallocgc` (аллокации памяти).

### Когда НЕ стоит использовать?

*   Для хранения соединений с базой данных или сетевых сокетов (для этого лучше использовать специализированные пулы соединений).
*   Для объектов, которые живут долго.
*   Если создание объекта обходится дешевле, чем логика работы с пулом (например, для маленьких структур из пары полей).

### Deadlock

**Дедлок** - это ситуация, когда все горутины находятся в спящем состоянии и не могут быть "разбужены". Пример:

```Golang
package main

func main() {
    // создаем канал
	c := make(chan int)

    // пытаемся получить данные с канала, которые отсутствуют. Deadlock!
	<-c
}
```

### Пакет "sync"

- **WaitGroup** - это механизм синхронизации, который позволяет дождаться завершения выполнения всех горутин, запущенных в рамках программы. _Для JavaScript разработчиков этот метод может быть схожим с `Promise.allSetteled`._

- **errgroup.WithContext** — устанавливается сторонним пакетом Go. Позволяет также дождаться завершения выполнения всех горутин, запущенных в рамках программы, только если одна горутина падает, то и выполнение всех остальных горутин тоже становится невозможным. _Для JavaScript разработчиков этот метод может быть схожим с `Promise.all`._

- **Mutex** - это механизм синхронизации, который позволяет управлять доступом к общим ресурсам. Он используется для предотвращения состояния гонки, когда несколько горутин пытаются изменить одни и те же данные одновременно.

- **Once** - это механизм синхронизации, который позволяет выполнить определенную функцию только один раз.

- **RWMutex** — это механизм синхронизации, который позволяет множеству горутин читать значение, не блокируя переменную. Пример:

```Golang
// я могу смело использовать RLock() для счетчика, так как он не меняет данные
func count() {
	rw.RLock()         // <-- заметьте букву R в RLock (read-lock)
	defer rw.RUnlock() // <-- заметьте букву R в RUnlock()
	return len(sharedState)
}

// Но я должен использовать Lock() для set(), который меняет данные
func set(key string, value string) {
	rw.Lock()                // <-- заметьте, тут мы берем "обычный" Lock (write-lock)
	defer rw.Unlock()        // <-- а тут Unlock(), без R
	sharedState[key] = value // <-- изменяет состояние(данные)
}
```

### Пакет "atomic"

Атомарные операции - это низкоуровневые примитивы синхронизации, которые обеспечивают способ выполнения операций чтения-модификации-записи (read-modify-write) над общими переменными без необходимости в блокировке.

Пакет "sync/atomic" предоставляет несколько атомарных операций для целых чисел (int), указателей и чисел с плавающей точкой(float) и свапом (swap). Пример:

```Golang
func main() {
    // объявляем счетчик
    var counter int64

    // создаем WaitGroup
    wg := sync.WaitGroup{}

    // цикл с 5-ю итерациями
    for i := 0; i < 5; i++ {
        // добавляем в список ожидания одну горутину
        wg.Add(1)
        go func() {
            // говорим, чтобы в конце анонимной функции одна горутина из списка исчезла
            defer wg.Done()
            // атомарной операцией добалвяем к счетчику единицу
            atomic.AddInt64(&counter, 1)
            fmt.Println("Счетчик:", atomic.LoadInt64(&counter))
        }()
    }
    // ожидаем незавершившиеся горутины
    wg.Wait()
    fmt.Println("Итоговый счетчик:", counter)
}
```

В данном случае нам не нужно использовать мютекс для защиты счетчика, так как атомарные операции обеспечивают его безопасное обновление.

## <a name="con_vs_parall_multi"></a>Конкурентное, параллельное, многопоточное и асинхронное исполнение

[Перед прочтением запомните: **Concurrency is not parallelism** (с) Rob Pike](https://go.dev/blog/waza-talk)

- **Конкурентность (concurrency)** - выполнение задач **ЗА** определённое время. необязательно выполняются одновременно, поэтому их можно разделить на более мелкие и чередующиеся.
  
  Проведём аналогию: шеф-повар нарезает лук и иногда проверяет духовку. Ему нужно прекратить нарезать лук, чтобы подойти к духовке, а затем снова начать нарезать и повторять этот процесс до конца приготовления блюда.

- **Параллельное исполнение (parallel computing)** — выполнение задач **В ОДНО** и то же время. Само название подразумевает, что они выполняются параллельно.

  Вернёмся на кухню: теперь у нас есть два шеф-повара. Один следит за духовкой, а второй нарезает лук. Работа разделена, так как теперь на кухне трудятся два повара.

- **Многопоточность** - один из способов реализации **конкурентного исполнения** путем выделения потока (worker thread).

- **Асинхронность (asynchrony)** — подразумевает, что операция может быть выполнена кем-то на стороне: удаленным веб-узлом, сервером или другим устройством за пределами текущего вычислительного устройства.

## <a name="scheduler"></a>Планировщик в Go

Как и в случае с планировщиком OS, мы не можем предсказать работу планировщика Go, так как он также недетерменирован. Все зависит от runtime Go.

Планировщик работает по GMP модели:

- **G (goroutine)** - очередь горутин (GRQ).

- **M (machine thread)** - поток OS;

- **P (processor)** - Это условный контекст, который объединяет **поток ОС (M)** и **очередь горутин (G)**. По умолчанию количество P берётся из значения переменной среды `GOMAXPROCS` и равно количеству логических ядер компьютера;

## <a name="make_new"></a>make и new

**`new`** — это встроенная функция для выделения памяти. Однако, в отличие от аналогичных функций в других языках, она не инициализирует выделенную память, а просто ставит в нее значение 0. Функция `new(T)` выделяет память для новой переменной типа `T`, ставит в нее значение 0 и возвращает адрес этой памяти – значение типа `*T`. С точки зрения терминологии языка Go, эта функция возвращает указатель на память, содержащую значение 0, принадлежащую переменной типа `T`.

```Golang
func main() {
	// Выделяем память под int. 
	// p — это указатель (тип *int)
	p := new(int) 

	// Проверяем, что лежит по этому адресу (используем звездочку * для разименования)
	fmt.Println(*p) // Выведет: 0 (память автоматически обнулена)

	// Теперь мы можем изменить значение по этому адресу
	*p = 42
	fmt.Println(*p) // Выведет: 42
}
```

При объявлении через `new` или `var`, поля структуры будут обнулены, то есть нулевыми, то есть у каждого поля в зависимости от типа будет нулевое значение:

```Golang
type SyncedBuffer struct {
  lock sync.Mutex
  buffer bytes.Buffer
}

p := new(SyncedBuffer) // тип *SyncedBuffer
var v SyncedBuffer // тип SyncedBuffer
```

---

Встроенная функция `make(T, args)` выполняет функцию, отличную от функции `new(T)`. Она создает объекты типа “слайсы”, “карты” и “каналы”. При этом функция возвращает значение типа `T` (а не `*T` ). Причина такого различия заключается в том, что все эти типы на самом деле представляют собой ссылки на структуры данных, которые необходимо сначала инициализировать перед использованием. Например, слайс представляет собой структуру, содержащую указатель на данные, находящиеся в массиве, а также информацию о длине и объеме этого массива. До момента инициализации этих данных слайс считается недействительным. Функция make отвечает за инициализацию внутренних структур данных и подготовку их к использованию.

```Golang
b := make(chan int)  // b имеет тип chan int
```

## <a name="interface"></a>Интерфейсы

Интерфейсны — абстракция, которая описывает поведение других типов.

При создании переменной интерфейса (не пустого, а именованный интерфейс с методами), под капотом это структура с двумя полями: указатель на значение и указатель на тип.

При вызове метода, гошка в рантайме по таблице находит место в памяти, где этот метод хранится и вызывает его. Причем, первый вызов будет подольше, а следующие вызовы уже быстрее из-за кэширования.

_Когда вы видите в Go объявление (переменной, параметра функции или поля структуры), имеющее интерфейсный тип, вы можете использовать объект любого типа, пока он удовлетворяет интерфейсу._

```Golang
// Объявляем тип Book, который удовлетворяет интерфейсу fmt.Stringer.
type Book struct {
    Title  string
    Author string
}

func (b Book) String() string {
    return fmt.Sprintf("Book: %s - %s", b.Title, b.Author)
}

// Объявляем тип Count, который удовлетворяет интерфейсу fmt.Stringer.
type Count int

func (c Count) String() string {
    return strconv.Itoa(int(c))
}

// Объявляем функцию WriteLog(), которая берёт любой объект,
// удовлетворяющий интерфейсу fmt.Stringer в виде параметра.
func WriteLog(s fmt.Stringer) {
    log.Println(s.String())
}

func main() {
    // Инициализируем объект Book и передаём в WriteLog().
    book := Book{"Alice in Wonderland", "Lewis Carrol"}
    WriteLog(book)

    // Инициализируем объект Count и передаём в WriteLog().
    count := Count(3)
    WriteLog(count)
}
```

## <a name="context"></a>Контекст (Context)

Контекстом называют интерфейс `Context` из пакета `context`.

Контекст используют для:

- чтобы устанавливать дедлайн исполнения блока кода;

- оповещать об окончании исполнения блока кода;

- узнавать причину отмены контекста;

- получать значения по ключу.

### Корневой контекст

Корневой контекст невозможно отменить, он не имеет дедлайна и не содержит значений.

_Обычно корневой контекст создается в main() и оборачивается в производный контекст с отменой, который затем передается в функции._

_Когда контекст отменяется в main(), отменяется и код в глубине стека вызовов. Это помогает реализовать корректное завершение работы процесса._

### Создание нового контекста

Все контексты, за исключением корневых, создаются на основе родительских.

Для создания новых контекстов в пакете предусмотрены функции `WithCancel`, `WithDeadline`, `WithTimeout`, `WithValue` и `WithoutCancel`.

### WithCancel

```Golang
ctx, cancel := context.WithCancel(context.Background())
defer cancel()
```

Функцию отмены нужно вызывать на том же уровне, где она была создана. Это называется отменой вручную. Вызывать функцию отмены в другом месте в Go считается антипаттерном, поскольку это может приводить к утечке незакрытых контекстов.

### WithDeadline и WithTimeout

```Golang
func slowOperationWithTimeout(ctx context.Context) (Result, error) {
	ctx, cancel := context.WithTimeout(ctx, time.Second)
	defer cancel()
	return slowOperation(ctx)
}
```

Функция `context.WithTimeout` принимает два аргумента: существующий контекст и `time.Duration`. Контекст будет отменен по истечении таймаута автоматически, но также обязательно отменять контекст вручную при помощи `defer`, чтобы не допустить утечки ресурсов (памяти и горутин).

### WithValue

```Golang
type favContextKey string

key := favContextKey("language")
ctx := context.WithValue(context.Background(), key, "Go")
```

Функция принимает три аргумента: родительский контекст, ключ и значение. Во избежание коллизий ключ не должен быть одним из встроенных типов и должен поддерживать операторы сравнения == и !=.

```Golang
ctxValue := ctx.Value(key)
```

Получить значение можно методом Value. Он принимает ключ и возвращает его значение. Если ключ не был найден, возвращается nil.

### WithoutCancel

`WithoutCancel` возвращает копию родительского контекста, которая не будет отменена при отмене родительского контекста.

### Обработка ошибок

Узнавать об отмене контекста помогают методы `Done` и `Err` экземпляра `context.Context.`

Метод `Err` может возвращать три варианта значений:

- `nil` — если контекст активен;

- `context.Canceled` типа error в случае явной отмены контекста;

- `context.DeadlineExceeded` в случае отмены по истечении времени.

## <a name="defer"></a>defer

Как работает defer в Go:

- Ключевое слово `defer` используется для отложенного вызова функции. Функция с `defer` всегда выполняется перед выходом из внешней функции, в которой defer объявлялась. `defer` добавляет переданную после него функцию в стэк, при возврате внешней функции вызываются все добавленные в стэк (LIFO — last in first out) вызовы.

```Golang
func main() {
	fmt.Println("counting")

	for i := 1; i < 4; i++ {
		defer fmt.Println(i)
	}

	fmt.Println("done")
}
```

Вывод:

```Golang
counting
done
3
2
1
```

- Аргументы функции, вызываемой через `defer`, вычисляются сразу при вызове `defer`, а сама функция, как показывалось выше, выполняется позже.

```Golang
func trace(s string) string {
  fmt.Println("entering:", s)
  return s
}

func un(s string) {
  fmt.Println("leaving:", s)
}

func a() {
  defer un(trace("a"))
  fmt.Println("in a")
}

func b() {
  defer un(trace("b"))
  fmt.Println("in b")
  a()
}

func main() {
  b()
}
```

Вывод:

```Golang
entering: b
in b
entering: a
in a
leaving: a
leaving: b
```

## <a name="errors"></a>Ошибки

В стандартной библиотеке ошибка — это интерфейс с одним методом, который возвращает строку:

```Golang
type error interface {
	Error() string
}
```

Этот `error` интерфейс можно крутить и дополнять так, как душе захочется:

```Golang
type MyError struct{}

func (m *MyError) Error() string {
	return "boom"
}

func sayHello() (string, error) {
	return "", &MyError{}
}

func main() {
	s, err := sayHello()
	if err != nil {
		fmt.Println("unexpected error: err:", err)
		os.Exit(1)
	}
	fmt.Println("The string:", s)
}
```

Или так:

```Golang
package main

import (
	"errors"
	"fmt"
	"os"
)

type RequestError struct {
	StatusCode int

	Err error
}

func (r *RequestError) Error() string {
	return fmt.Sprintf("status %d: err %v", r.StatusCode, r.Err)
}

func doRequest() error {
	return &RequestError{
		StatusCode: 503,
		Err:        errors.New("unavailable"),
	}
}

func main() {
	err := doRequest()
	if err != nil {
		fmt.Println(err)
		os.Exit(1)
	}
	fmt.Println("success!")
}
```

## <a name="concurrency_patterns"></a>Паттерны многопоточности

### Future/Promise

Паттерн **Future/Promise** хорошо подходит для тех случаев, когда ты хочешь запустить задачу в фоне и получить ее результат, не блокируя выполнение программы. Это отличный способ обрабатывать долгие операции, такие как сетевые запросы или сложные вычисления и одновременно делать другие, полезные,вещи, не теряя при этом контроль над результатами.

Детали:

- **Promise** — обещание предоставить результат операции в будущем. Что-то, что выполняет это обещание.

- **Future** — это объект, который позволяет проверять готовность результата и извлекать его, когда он будет доступен.

### Generator

Генератор создает значения и отправляет их потребителю через канал. При этом отправка и получение данных блокируются до тех пор, пока обе стороны не будут готовы. Такой подход позволяет обрабатывать данные по мере их поступления, минимизируя задержки.

### Pipeline

Паттерн **Pipeline** отлично подойдет для задачи, где обработка может быть разбита на несколько этапов. Как вариант можно организовать конвеер для поиска данных, фильтрации и сохранения результатов в БД.

### Fan-In и Fan-Out

Паттерн **Fan-Out** позволяет распределить вычисления на несколько горутин, для параллельной обработки данных. Таким образом, ты можешь увеличить пропускную способность.

А что если потребуется потребуется слить эти каналы обратно? Для этого есть паттерн **Fan-In**. Он объединяет несколько каналов в один.

### Semaphore

Используется для того, чтобы ограничить количество горутин, которые одновременно могут работать с определенным ресурсом. Это помогает избежать перегрузки ресурса, например, базы данных, и позволяет управлять количеством параллельно выполняемых задач.

Он подходит для ситуаций, когда ресурсы, такие как база данных или API, могут обрабатывать ограниченное количество запросов одновременно.

### WorkerPool

Используется для паралельной обработки задач из общей очереди.

Основные элементы:

- Очередь задач, через которую воркеры получают задачи на выполнение.

- Набор воркеров, которые берут задачи из очереди и обрабатывают их.

**Worker Pool** отлично подходит для ситуаций, когда нужно обрабатывать большое количество однотипных задач, например, входящие запросы к API, работа с файлами, запросы к базе данных и другие задачи, требующие параллельной обработки. Паттерн позволяет эффективно распределять нагрузку, не создавая излишнего количества горутин.

## <a name="escape_analysis"></a>Как работает escape analysis и куда уходят переменные?

### Стек и куча: основы

- **Стек (stack)** — память, которая работает по принципу LIFO (last in, first out). Переменные живут до выхода из функции. Работа со стеком быстрая и почти бесплатная.

- **Куча (heap)** — область памяти для объектов, чьё время жизни нельзя заранее определить. Управляется сборщиком мусора (GC). Аллокации дороже, а сборка может замедлять программу.

**Главное отличие: на стеке память освобождается автоматически при выходе из функции, а в куче за этим следит GC.**

### Escape analysis: кто сбежал в кучу?

Когда компилятор Go встречает переменную, он решает, где её хранить — в стеке или куче. Этот процесс называется escape analysis.

Если переменная "сбегает" за пределы функции, она отправляется в кучу. Если нет — остаётся в стеке.

### Пример 1. Локальная переменная (остаётся в стеке)

```Golang
func sum(a, b int) int {
    c := a + b
    return c
}
```

### Пример 2. Переменная уходит в кучу

```Golang
func makePointer() *int {
    x := 42
    return &x
}
```

Здесь `x` возвращается по указателю. После выхода из функции стек очистится, и значение потерялось бы. Поэтому `x` "сбегает" в кучу, чтобы продолжить жить.

### Когда переменные убегают в кучу?

- Возвращаются из функции по указателю.

- Сохраняются в замыкании.

- Передаются в интерфейс (иногда).

- Размер структуры слишком большой, чтобы поместиться в стек.
